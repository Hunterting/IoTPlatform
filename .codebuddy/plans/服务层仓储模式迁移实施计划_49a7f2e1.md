## 产品概述

将IoT平台服务层从直接使用DbContext迁移到仓储模式，提升数据访问层的抽象性、可测试性和代码重用性。

## 核心功能

- 清理AlertService中的重复代码，完成仓储模式迁移
- 迁移高优先级服务：AuthService、UserService、RoleService
- 迁移中优先级服务：AreaService、DeviceService、MonitoringService、WorkOrderService
- 迁移辅助功能服务：ArchiveService、LogService、DictionaryService、SettingsService
- 迁移高级功能服务：ProtocolConfigService、DataRuleService、ETLTaskService、DataCollectionService
- 确保所有服务使用仓储接口和UnitOfWork进行数据访问和事务管理
- 使用AutoMapper进行对象映射，简化代码

## 技术要求

- 保持现有业务逻辑不变，仅改变数据访问方式
- 保持所有API端点功能正常
- 确保事务管理正确实现
- 维护多租户和区域权限过滤机制

## 技术栈

- 后端框架：.NET 8.0
- ORM框架：Entity Framework Core
- 仓储模式：自定义仓储接口和实现（已完成）
- 对象映射：AutoMapper
- 依赖注入：内置DI容器

## 技术架构

### 系统架构

采用三层架构模式：表现层（Controllers）→ 业务逻辑层（Services）→ 数据访问层（Repositories）。服务层不再直接依赖DbContext，而是通过仓储接口访问数据。

### 架构转换流程

```
当前架构：
Controller → Service (使用DbContext) → Entity Framework

目标架构：
Controller → Service (使用Repository接口) → Repository实现 → Entity Framework
                         ↓
                   UnitOfWork（事务管理）
```

### 模块划分

- **仓储层**：18个已实现的仓储类（IRepository<T>通用仓储+专用仓储）
- **服务层**：20个业务服务类（需迁移）
- **工作单元**：UnitOfWork统一管理事务

### 数据流

1. 用户请求到达Controller
2. Controller调用Service方法
3. Service通过仓储接口查询/操作数据
4. Service通过UnitOfWork保存更改和事务管理
5. Service使用AutoMapper映射实体到DTO
6. 返回结果到Controller

## 实现方法

### 迁移策略

1. **分批次迁移**：按优先级分4个阶段完成，降低风险
2. **保持功能对等**：迁移前后业务逻辑完全一致
3. **重构AlertService**：先清理重复代码，确保其完全使用仓储模式
4. **参考示例**：以AlertService的仓储部分为模板迁移其他服务

### 关键技术决策

1. **依赖注入**：服务类构造函数注入IUserRepository、IUnitOfWork、IMapper等接口
2. **事务管理**：使用UnitOfWork.BeginTransactionAsync/CommitTransactionAsync/RollbackTransactionAsync
3. **查询优化**：优先使用仓储的专用方法，复杂查询使用仓储的Query方法
4. **对象映射**：使用_mapper.Map<DtoType>(entity)简化映射代码
5. **权限过滤**：利用仓储层的自动AppCode过滤和AllowedAreaIds参数

### 性能与可靠性

- **复杂度**：CRUD操作O(1)到O(n)，分页查询O(n)，已通过仓储层优化
- **瓶颈**：N+1查询问题通过仓储的Include和预加载解决
- **缓解策略**：批量操作使用仓储的批量方法，减少数据库往返
- **技术债务**：不改变现有架构，仅迁移数据访问层，避免过度设计

## 实现注意事项

### 代码变更控制

- **性能**：确保迁移后查询性能不下降，利用仓储层的查询优化
- **日志**：保持现有日志记录方式，不引入新日志调用
- **爆炸半径**：逐个服务迁移，每个服务独立测试，失败可回滚
- **向后兼容**：Service接口签名不变，Controller无需修改

### 特殊场景处理

1. **多表关联查询**：使用仓储的Include方法或专用查询方法
2. **复杂过滤条件**：使用仓储的Query方法配合表达式树
3. **事务操作**：使用UnitOfWork的Begin/Commit/Rollback方法
4. **登录日志**：AuthService中的RecordLoginLogAsync使用仓储的LogRepository
